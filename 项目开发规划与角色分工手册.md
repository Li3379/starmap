# 星图 StarMap —— 项目开发规划与角色分工手册

> 配套文档：《星图-项目设计文档 v2.0》
> 适用：9 人团队 × 15 周开发周期
> 目标：闭环开发、角色解耦、按周可追踪、飞书同步

---

## 第一部分：9 人角色定义与解耦边界

> **核心原则**：每个人只对自己的"输入契约"和"输出契约"负责，不依赖其他人的实现细节。

### 1.1 角色总表

| 编号 | 角色 | 昵称占位 | 工作流 | 人数 | 核心交付物 |
|------|------|---------|--------|------|-----------|
| R0 | 技术负责人/架构师 | 待填 | 跨流 | 1 | 架构、决策、契约签字、风险统筹 |
| R1 | 后端开发-数据岗 | 待填 | 流A | 1 | 爬虫、清洗、入库 |
| R2 | 后端开发-服务岗 | 待填 | 流A | 1 | 图谱服务、API、Docker |
| R3 | 算法开发-抽取岗 | 待填 | 流B | 1 | 信息抽取、归一化、Prompt |
| R4 | 算法开发-演化岗 | 待填 | 流B | 1 | 信任度、幻觉防控、演化、匹配 |
| R5 | 前端开发-图谱岗 | 待填 | 流C | 1 | 全景图谱页、岗位详情页 |
| R6 | 前端开发-应用岗 | 待填 | 流C | 1 | 匹配诊断页、质量看板、管理后台 |
| R7 | QA 与评估 | 待填 | 流D | 1 | 评估方法学、Golden Set、评分脚本、准确率达标 |
| R8 | 文档与材料统筹 | 待填 | 跨流 | 1 | 设计文档、PPT、视频、提交包 |

> ⚠️ 后端组文档 §11.2 写"3人"，但 §13.2 角色矩阵写"2人"——本规划按 §13.2 的 **2人** 执行（R1+R2），技术负责人 R0 跨流统筹不归属后端组。

### 1.2 每个角色的"输入/输出契约"（解耦关键）

| 角色 | 输入（上游交付给我）| 输出（我交付给下游）| 不依赖（解耦边界）|
|------|-------------------|-------------------|------------------|
| **R0 技术负责人** | 各流进度看板、风险反馈 | 架构决策（附录D）、签字、资源调配 | 不写业务代码 |
| **R1 后端-数据** | 爬虫目标清单（R0定）| `jd_raw` 表数据、compliance_log | 不依赖 R3 抽取结果 |
| **R2 后端-服务** | 契约（R3+R0定）、本体（R3定）| REST API（符合 openapi.yaml）、Docker 镜像 | 不依赖 R5/R6 前端 |
| **R3 算法-抽取** | JD 原文（R1供）、契约 schema | extraction_result 表、抽取 Prompt | 不依赖 R4 演化代码 |
| **R4 算法-演化** | extraction_result（R3供）、图谱（R2供）| 信任度评分、演化检测结果、匹配结果 | 不依赖 R1 爬虫细节 |
| **R5 前端-图谱** | 契约（R3定）、mock 数据（自己造）| 全景图谱页、岗位详情页 | 不依赖 R2 API 实现 |
| **R6 前端-应用** | 契约、mock 数据 | 匹配诊断页、质量看板、管理后台 | 不依赖 R4 算法实现 |
| **R7 QA** | Golden Set 真值（自标）、系统输出（R3/R4供）| 评分报告、准确率通报 | 完全独立，全程并行 |
| **R8 文档** | 各流交付物、决策记录 | 设计文档、PPT、视频、提交包 | 不写代码，只整合 |

> **解耦的精髓**：R5/R6 前端用 mock 数据先行，W6 才联调；R7 评估全程独立。这样任何一个人延期，都不会阻塞其他人。

## 第四部分：角色解耦的具体执行机制

### 4.1 "契约先行"如何落地

```
W1-W2 流程：
R3 编写 openapi.yaml → R0 评审签字 → 
R2 后端按契约实现 API → R5/R6 前端 gen:api 生成类型 + MSW mock
```

**关键点**：契约签字前，R2 不写业务代码，R5/R6 用 mock 先行。签字后大家并行开发。

### 4.2 数据流转的解耦

```
R1 爬虫 → jd_raw 表 ──┐
                       ├→ R3 抽取 → extraction_result 表 → R4 演化/匹配
ESCO 本体 ─────────────┘                                        ↓
                                                           匹配结果表
                                                               ↓
R5/R6 前端 ←───────────────── R2 API ←──────────────────── 图谱查询
```

**关键点**：每个人只读写自己负责的表/接口，不跨边界改别人的数据。

### 4.3 评估的独立性

```
R7 QA 全程独立：
- 自己标注 Golden Set（真值保密，D4决策）
- 接收 R3/R4 的系统输出
- 跑 evaluate.py 评分
- 报告准确率给 R0

不参与算法调优，避免"既当运动员又当裁判"
```
